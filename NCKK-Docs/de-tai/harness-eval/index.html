1. Introduction

1.1. Problem Statement

Large Language Model (LLM)-based coding agents such as Claude Code, GitHub Copilot, Cursor, and SWE-Agent have become integral tools in modern software development. These agents autonomously read source code, fix bugs, generate features, and interact with file systems, terminals, and version control systems [1, 16].

However, the performance of a coding agent depends on two distinct factors: (1) the underlying LLM (e.g., Claude, GPT, DeepSeek) and (2) the harness—the infrastructure scaffold comprising the tool dispatch system, context management layer, and supporting components [1]. Lou et al. [2] demonstrated that the harness plays a critical role: a smaller model with a well-designed harness can outperform a larger model without one, eliminating 100% of illegal actions in constrained environments. Bui et al. [1] described the harness as an "operating system for the agent," featuring a multi-layer architecture including dual-agent design, Adaptive Context Compaction (ACC) that reduces token usage by 54%, and event-driven system reminders to combat instruction fade-out.

The current paradox. The research community has developed numerous benchmarks for evaluating agent output—SWE-Bench Pro with 1,865 tasks [5], SWE-Compass with 2,000 instances across 10 languages [9], and PRDBench with agent-driven evaluation [8]. However, an analysis of 49 recent papers (2023–2026) reveals that no study systematically evaluates the harness itself [19]. When reporting "Agent A achieves 70% resolve rate," we do not know which harness component contributed to that result.

Why this is a genuine research gap. A legitimate question is: why have the groups that develop harnesses (Princeton—SWE-Agent, Anthropic—Claude Code) not conducted this evaluation themselves? Three reasons explain this gap:

  1. Different roles. They are developers with incentives to build new harnesses, not evaluators with incentives to compare systems—analogous to how automobile manufacturers build cars while independent organizations (Euro NCAP, IIHS) conduct safety evaluations.
  2. Rapid field growth. From 2023 to 2026, the number of coding agents grew from a handful to dozens. No group has paused to meta-evaluate because the field is in a construction race.
  3. Indirect evidence of need. Lou et al. [2] compared harnesses on game tasks; Robeyns et al. [4] self-improved a harness without measuring individual component contributions—demonstrating that the need exists but has not been formalized.

Research gap. There is no evaluation framework that enables: (a) measuring the quality of individual harness components, (b) decomposing the contributions of harness vs. LLM, and (c) comparing different harnesses across unified quality dimensions.

1.2. Research Purpose (Qualitative)

This study aims to construct the first integrated evaluation framework for coding agent harness infrastructure, enabling the decomposition and quantification of individual harness component contributions to overall agent performance, independently of the underlying LLM capability.

1.3. Research Objectives (Quantitative)

RO1. Design a 3-dimensional harness evaluation taxonomy (Tool Dispatch Efficiency, Context Utilization, Backend Portability) with 7 metrics total, validated by at least 5 domain experts (inter-rater agreement measured by Fleiss' κ ≥ 0.6).
RO2. Fork and refactor SWE-Agent into a modular harness with swappable components (tool set, context strategy, model backend), ensuring resolve rate equivalent to baseline SWE-Agent (±3% absolute) on 50 verification tasks.
RO3. Conduct a systematic ablation study on 150 tasks from SWE-Bench Verified with 27 conditions (3×3×3 full factorial: tool configurations × context strategies × LLM backends), including a pilot study (5 conditions × 20 tasks) prior to the full experiment. Report mean ± standard deviation from 3 runs for the 10 most critical conditions.
RO4. Decompose variance contributions of harness vs. LLM using three-way ANOVA (Tool × Context × Backend), with statistical significance at p < 0.05 after Bonferroni correction. Include a Generalized Linear Mixed Model (GLMM) as a robustness check for binary outcomes.
RO5. Release an open-source evaluation toolkit on GitHub with documentation, examples, and CI/CD pipeline.

1.4. Research Hypotheses

H1. The tool system has the largest effect size (Cohen's d > 0.5) among the three harness components on resolve rate.
H2. Context management has a medium effect size (Cohen's d = 0.3–0.5).
H3. Harness configuration explains at least 10% of resolve rate variance (η² ≥ 0.10 in ANOVA), independent of LLM backend.
H4. Resolve rate variance across LLM backends decreases when harness quality is higher (statistically significant interaction effect).

1.5. Scope and Delimitations

In scope:

  • Harness/scaffold infrastructure of LLM-based coding agents
  • Three harness components: tool system, context manager, model backend
  • 150 tasks from SWE-Bench Verified (Python)
  • Three LLM backends: Claude Sonnet 4, GPT-4o, DeepSeek-V3
  • Base harness: SWE-Agent (forked and refactored)

Out of scope (future work):

  • Safety/security evaluation → separate study (SecureCodeBench)
  • Session continuity / long-horizon evaluation
  • Languages other than Python
  • Commercial harnesses (Claude Code, Cursor)

2. Literature Review and Theoretical Background

2.1. Literature Review

This section reviews the five works most closely related to our study. Each addresses a subset of harness evaluation concerns but leaves critical gaps, which are summarized in Table 1.

2.1.1. Bui et al. (OpenDev) [1]: Harness Design Without Evaluation

Bui et al. [1] provided the most detailed harness description to date: a terminal-native coding agent with dual-agent architecture (Planner-Executor), 5-stage Adaptive Context Compaction (ACC) reducing token usage by 54%, lazy tool discovery, and event-driven system reminders to combat instruction fade-out. The paper introduced the term "harness" as an analogy to an operating system for agents and described a four-layer architecture with safety sandboxing.

What is present (△ in C1, C2): The paper describes tool dispatch logic (which tools to expose, when to load them) and context management strategies (ACC stages, reminder injection). However, these descriptions are qualitative design choices, not quantitative evaluations—no metric measures how effective the tool dispatch is, nor how much information ACC retains after compaction.

What is missing (_ in C3, C4, C5, C6): OpenDev was tested with a single LLM backend, so cross-backend portability is unknown (C3). No component was systematically removed to measure its individual contribution (C4). The paper reports overall resolve rate but does not decompose how much is attributable to the harness vs. the underlying LLM (C5). There is no multi-metric evaluation framework—only resolve rate (C6).

2.1.2. Lou et al. (AutoHarness) [2]: Ablation on Games, Not Coding

Lou et al. [2] demonstrated that LLMs can automatically synthesize code harnesses through iterative refinement with environment feedback. Their key finding—that a smaller model with an auto-generated harness outperforms a larger model without one—provides strong evidence that harness quality matters independently of model size.

What is present (△ in C4): AutoHarness performs a form of ablation: comparing agent performance with vs. without a harness. This is the closest existing work to our ablation methodology. However, the evaluation is conducted on game environments (TextArena, 145 games), not on software engineering tasks, and the "ablation" is binary (harness present/absent) rather than systematic (varying individual components).

What is missing (_ in C1, C2, C3, C5, C6): Tool dispatch and context management are not evaluated separately (C1, C2). Only one LLM backend is used per experiment (C3). There is no variance decomposition between harness and model (C5). No multi-metric framework is proposed (C6).

2.1.3. Qin et al. (ToolLLM) [12]: Tool Evaluation in Isolation

Qin et al. [12] introduced ToolBench, a benchmark of 16,464 real-world REST APIs, and ToolLLaMA, a model fine-tuned to match ChatGPT-level tool-use performance. They proposed ToolEval with automatic metrics for tool selection and task completion.

What is present (✓† in C1): ToolLLM provides the most comprehensive tool evaluation to date, with metrics for API selection accuracy and multi-step tool planning via DFSDT (Depth-First Search-based Decision Tree). However, tools are evaluated in isolation—each API call is independent, with no interaction with context management, file systems, or coding-specific workflows. The benchmark tests whether an LLM can call the right API, not whether a harness dispatches tools efficiently within a multi-turn coding session.

What is missing (_ in C2, C3, C4, C5, C6): Context management is not assessed (C2). The benchmark uses a fixed model, so portability is untested (C3). No ablation is performed on harness components (C4). Model vs. infrastructure contributions are not separated (C5). The evaluation is single-metric (pass rate) (C6).

2.1.4. Chhikara et al. (Mem0) [10]: Context Evaluation on Chatbots

Chhikara et al. [10] introduced Mem0, a scalable memory architecture that dynamically extracts, consolidates, and retrieves information across conversations. It achieves 91% lower p95 latency and over 90% token cost savings compared to full-context approaches, with a graph-based variant (Mem0g) that captures entity relationships.

What is present (✓* in C2): Mem0 provides rigorous context/memory evaluation with metrics for retrieval accuracy, latency, and token efficiency. However, evaluation is conducted exclusively on the LoCoMo benchmark—a conversational QA dataset—not on coding tasks. In coding, "context" involves AST structures, file dependency graphs, test result histories, and multi-file relationships, which are structurally different from conversational memory.

What is missing (_ in C1, C3, C4, C5, C6): Tool dispatch is not evaluated (C1). Only one backend is used (C3). No component ablation (C4). No harness vs. LLM decomposition (C5). Single-domain evaluation (chatbot), no multi-metric coding framework (C6).

2.1.5. Ma et al. (AgentBoard) [15]: Multi-Metric for Output, Not Harness

Ma et al. [15] introduced AgentBoard, a benchmark with 9 diverse environments and 1,013 tasks. Its key contribution is the progress rate metric—a fine-grained measure of partial task completion that correlates with human judgment at Pearson > 0.95. AgentBoard evaluates agents across multiple dimensions (web browsing, tool use, game playing, etc.).

What is present (✓‡ in C6): AgentBoard is the most mature multi-metric evaluation framework. However, all metrics measure agent output quality (did the agent make progress toward the goal?), not harness infrastructure quality (did the harness dispatch tools efficiently? did it manage context well?). The framework evaluates what the agent achieved, not what the harness provided.

What is missing (_ in C3, C4, C5): Agents are tested on a fixed scaffold, so cross-backend portability is not measured (C3). No scaffold component is varied (C4). There is no decomposition of scaffold vs. model contributions (C5).

Reading the table. The table reveals two patterns that define our research gap:

  1. Columns C3, C4, C5 are entirely "_" across all five papers. No prior work has (C3) compared the same harness across multiple LLM backends, (C4) systematically ablated harness components in coding tasks, or (C5) decomposed variance between harness and LLM contributions.   2. Columns C1 and C2 have partial coverage (△ or qualified ✓) but never within a coding harness. ToolLLM evaluates tool accuracy on standalone APIs (✓†), not tool dispatch within a multi-turn coding agent session. Mem0 evaluates context on chatbot conversations (✓*), not on coding tasks involving AST structures and test histories. OpenDev and AgentBoard partially address these (△) but without formal metrics. HarnessEval bridges this gap by evaluating tool dispatch and context utilization within an integrated coding harness using formal, validated metrics (M1.1–M2.2).

In summary, existing works either (a) describe harness designs without evaluating them [1], (b) evaluate components in isolation outside the harness context [12, 10], (c) perform ablation on non-coding domains [2], or (d) provide multi-metric frameworks that measure output rather than infrastructure [15]. HarnessEval is the first to unify all six characteristics into a single evaluation framework for coding agent harnesses.

2.2. Theoretical Background

2.2.1. Harness Architecture: Five-Layer Model

Based on [18] and [1], a coding agent harness comprises five layers:

Layer 5 — Orchestration: Agent loop (ReAct, Plan-Execute), multi-turn management (future work)
Layer 4 — Safety & Permission: Sandboxing, permission model, input validation (future work)
Layer 3 — Context & Memory Management: Context window management, compaction, persistent memory EVALUATED (D2)
Layer 2 — Tool System: Available tools (Read, Write, Edit, Bash, Grep, Glob, etc.) and dispatch logic EVALUATED (D1)
Layer 1 — Model Backend: The underlying LLM (Claude, GPT, DeepSeek) EVALUATED (D3)

This study evaluates Layers 1–3. Layers 4–5 are designated as future work.

2.2.2. Ablation Studies in Software Engineering

Ablation study is a standard methodology in machine learning: systematically removing or modifying components to measure their individual contributions. Applied to harness evaluation:

  • Modify tool set (12 → 8 → 5 tools) → measure resolve rate change
  • Change context strategy (full → sliding window → summary) → measure resolve rate change
  • Switch LLM backend (Claude → GPT → DeepSeek) → measure portability

Confounding variables. When removing a tool, performance may decrease because (a) the agent lacks the capability, or (b) the harness dispatches poorly. To address this: we reduce tools gradually rather than removing entirely, and record which tasks require the removed tool to distinguish the two causes.

2.2.3. Defining "Independent of LLM" via Three-Way ANOVA

"Independent of LLM" does not mean the harness and LLM do not interact. It means their contributions can be statistically decomposed.

Method: Three-way ANOVA (following reviewer feedback to use 3-way rather than 2-way):

  • Factor A: Tool Configuration (3 levels)
  • Factor B: Context Strategy (3 levels)
  • Factor C: LLM Backend (3 levels)
  • Dependent variable: Resolve rate
Total Variance = VTool + VContext + VBackend + VT×B + VC×B + VT×C + VT×C×B + VError (1)
 VTool + VContext  = Harness     VT×B + VC×B  = Interactions

If VTool and VContext are statistically significant, the harness has an independent effect. The effect size η² indicates the percentage of variance explained by each factor. This formalization is the primary theoretical contribution of this study.

Robustness check: Since resolve rate per task is binary (0/1), ANOVA assumptions may be violated. A Generalized Linear Mixed Model (GLMM) with binary outcome and random effects for tasks will be used as a robustness check. Both ANOVA (for interpretability) and GLMM (for statistical correctness) will be reported.

3. Research Methodology

3.1. Overview

This study employs an experimental methodology with four phases:

  1. Taxonomy design and expert validation
  2. SWE-Agent fork and modular refactoring
  3. Pilot study (5 conditions × 20 tasks)
  4. Full ablation study (27 conditions × 150 tasks)

Validation procedure: The taxonomy will be presented to 5 domain experts (at least 2 from industry, at least 2 from academia). Each expert independently rates 20 sample evaluations. Inter-rater agreement will be measured using Fleiss' κ, with a target of κ ≥ 0.6.

3.3. Modular Harness via SWE-Agent Fork (RO2)

SWE-Agent is an open-source (MIT license) coding agent with approximately 15,000 LOC. We fork and refactor three modules:

  • tool_config: Swappable tool sets (12/8/5 tools) via configuration file
  • context_strategy: Swappable strategies (full/sliding-window/summary) via configuration file
  • model_backend: Swappable LLM backends (Claude/GPT/DeepSeek) via configuration file

Acceptance criteria: The modular harness must achieve resolve rate within ±3% absolute of baseline SWE-Agent on 50 verification tasks. The orchestration logic remains unchanged to ensure baseline quality.

Estimated effort: 2–3 weeks (vs. 1–3 months if built from scratch).

Table 4: Full Factorial Experimental Design
FactorLevelsDetails
Tool Configuration3Full (12 tools) / Medium (8) / Minimal (5)
Context Strategy3Full context / Sliding window (50K tokens) / Summary-based

Dataset: 150 tasks from SWE-Bench Verified, selected via stratified random sampling by difficulty (easy/medium/hard based on number of models that solved each task). Tasks with unstable Docker environments are excluded via pre-filtering.

Runs:

  • 10 most important conditions: 3 runs each (report mean ± std)
  • 17 remaining conditions: 1 run each
  • Total: (10 × 3 + 17 × 1) × 150 = 7,050 evaluations

Statistical analysis (RO4):

  • Three-way ANOVA: Tool (3) × Context (3) × Backend (3)
  • Report: F-statistic, p-value (Bonferroni-corrected), η² (eta-squared)
  • Post-hoc: Tukey HSD for pairwise comparisons
  • Effect size: Cohen's d for each pairwise comparison
  • Robustness check: GLMM with binary outcome (solved/not solved), random effects for tasks, fixed effects for Tool, Context, and Backend
  • Power analysis: with 150 tasks, 27 conditions, α = 0.05, power ≥ 0.80

3.6. Threats to Validity

3.6.1. Internal Validity

  • Confounding: Reducing tools may change the problem, not just harness quality. Mitigation: Reduce tools gradually (12→8→5); record which tasks require removed tools; exclude those from component analysis.
  • LLM non-determinism: Stochastic outputs introduce variance. Mitigation: Set temperature = 0; run 3 times for critical conditions; report confidence intervals.
  • Docker instability: Some SWE-Bench environments are unstable. Mitigation: Pre-filter tasks; retry twice on failure.
  • LLM versioning: Models may be updated during the study. Mitigation: Record exact model version/snapshot; run all experiments within a short time window.

3.6.2. External Validity

  • Python only: Tool systems for Python (pytest, pip) differ from Java (Maven, JUnit) or Rust (Cargo). Results require caution when generalizing.
  • Open-source only: Commercial harnesses (Claude Code, Cursor) may exhibit different behavior.
  • Survivor bias: Excluded tasks (Docker failures) may be harder tasks where harness matters more.

3.6.3. Construct Validity

  • M1.1 depends on annotator judgment. Mitigation: 2 annotators, Fleiss' κ.
  • M2.2 uses an LLM classifier. Mitigation: Validate on 200 segments with human labels; acceptance threshold: accuracy > 90%.
  • Statistical conclusion validity: Multiple testing across 7 metrics requires Bonferroni or Benjamini-Hochberg FDR correction.

4. Expected Results

4.1. Expected Outcomes per Objective

RO1 — Taxonomy: A validated 3-dimensional, 7-metric taxonomy with Fleiss' κ ≥ 0.6.

RO2 — Modular Harness: A fork of SWE-Agent with modular components, resolve rate within ±3% of baseline.

RO3 — Ablation Study: Empirical results answering:

  • Which component contributes most to resolve rate? (H1: Tool system has largest effect)
